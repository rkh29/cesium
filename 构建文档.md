# 卫星智能运维 Agent 构建文档

## 1. 目标

本次建设目标是在现有前端项目中落地一个可直接演示、可继续扩展的“卫星智能运维 Agent 控制台”，用于承接以下能力：

- 面向 AetherMind-450 风格卫星网络的异常感知、诊断、决策与处置展示
- 面向地面 L1/L2、多星 L3/L4 的分层 Agent 协作编排
- 面向后续异常检测算法模块的可插拔接入
- 面向千问大模型的实际 API 接入与调试
- 面向审批、黑板、工作流、联邦诊断的统一前端入口

本次实现范围优先覆盖前端控制台、Mock 编排闭环、Qwen 接入通道和后续算法接口预留，不把文档中尚未验证的“全量在轨自治能力”写成已完成事实。

提炼出的关键约束如下：

- 系统不是单一聊天机器人，而是“异常检测 + 运维决策 + 工作流 + 审批 + 黑板归档”的工程系统
- Agent 应采用分层协作结构，而不是简单单体问答
- 地面侧适合采用“Coordinator + Specialist”的受控编排
- 星上侧适合采用边缘快速检测、组级汇聚、全局协同的分级治理
- 高风险动作必须保留审批闭环
- 大模型输出应服务于结构化任务，而不是替代所有规则逻辑
- 后续异常检测算法会成为系统输入之一，因此前端和 Mock 服务必须预留算法输出接口

## 3. 落地架构

### 3.1 总体分层

本项目的前端侧采用四层落地：

1. 展示层
   负责智能运维控制台、Trace、Blackboard、审批、模型配置、联邦协商可视化。
2. 编排接入层
   负责调用 `/api/agents`、`/api/agents/chat`、`/api/agents/coordinator/chat`、`/api/blackboard`、`/api/approval`、`/api/llm` 等接口。
3. 业务语义层
   负责把“对话结果”转换成“运维上下文”，包括任务受理、Trace 跟踪、审批决策、黑板状态展示。
4. 算法接口层
   为后续异常检测模块预留统一接入面，支持注入“异常事件流”“根因评分”“置信度”“处置建议”等数据。

### 3.2 Agent 角色映射

前端页面与文档中的 Agent 分层保持一致，但在工程上简化成以下可展示对象：

- `Coordinator`
  对应 L1 总指挥，负责接收用户指令、生成排障意图、创建 trace。
- `Network Specialist`
  对应 L2 网络优化专家，聚焦链路、流表、网关切换、A2A 协商。
- `Health Specialist`
  对应 L2 异常处理专家，聚焦硬件、热控、电源、姿轨和遥测分析。
- `Ops Specialist`
  对应 L2 运维执行专家，聚焦工作流、恢复动作和操作建议。
- `Edge Agents`
  对应 L4 边缘异常检测节点，展示其心跳和分布，不在前端直接做自治。

### 3.3 参考 Claw Code 的编排思想

参考 `claw-code-main` 的核心并不是“照搬源码”，而是吸收以下工程思想：

- Coordinator 负责统一入口和任务拆解
- Specialist 负责受限职责域内的处理
- 子任务应尽量隔离上下文污染
- 审批、高危动作和工具调用应显式可见

因此本项目的前端不会做“自由聊天式多 Agent 幻觉界面”，而是做“聊天入口 + Trace + Blackboard + 审批 + 模型配置”的受控控制台。

## 4. 页面规划

### 4.1 页面名称

沿用现有路由 `/agent`，将 `AgentChat.vue` 升级为“卫星智能运维 Agent 控制台”。

### 4.2 页面模块

页面拆成以下区域：

- 顶部态势卡片
  展示 Coordinator、Specialist、Edge Agent、审批数量、联邦协商数量、LLM 在线状态。
- 左侧主交互区
  提供任务输入、快速指令、Agent 类型切换、聊天消息流、建议动作展示。
- 右侧运维上下文区
  展示当前 Trace、执行计划、Blackboard、待审批动作、A2A 协商记录、模型配置。
- 模型配置区
  可读取与更新 `/api/llm/config`，并执行连通性测试。

### 4.3 用户操作主路径

1. 用户输入运维问题，例如“排查 GEO 骨干丢包异常”
2. 前端并行调用：
   - `/api/agents/chat` 获取自然语言响应
   - `/api/agents/coordinator/chat` 创建或续接 trace
3. 前端根据返回的 `trace_id` 拉取：
   - `/api/agents/trace/:traceId`
   - `/api/blackboard/:traceId`
4. 页面同时展示：
   - 响应文本
   - Trace 计划
   - 黑板状态
   - 待审批动作
   - 联邦协商结果
5. 若存在高危动作，用户在审批区做通过/驳回

## 5. API 设计与复用策略

本项目已有较完整 API 结构，因此优先复用，不重新发明一套协议。

### 5.1 已复用接口

- Agent 状态：`GET /api/agents`
- 对话：`POST /api/agents/chat`
- 协调受理：`POST /api/agents/coordinator/chat`
- Trace：`GET /api/agents/trace/:traceId`
- Blackboard：`GET /api/blackboard/:traceId`
- 审批列表：`GET /api/approval/pending`
- 审批决策：`POST /api/approval/:requestId/decision`
- 联邦协商：`GET /api/applications/a2a/negotiations`
- 模型状态：`GET /api/llm/status`
- 模型配置：`GET/PUT /api/llm/config`
- 模型连通性测试：`POST /api/llm/test`

### 5.2 后续异常检测算法接入点

建议后续后端或算法服务至少补充以下统一接口之一：

方案 A：事件流接口

- `POST /api/anomaly/events`
- 输入字段建议：
  - `source`
  - `satellite_id`
  - `layer`
  - `event_type`
  - `severity`
  - `confidence`
  - `features`
  - `timestamp`

方案 B：根因诊断接口

- `POST /api/anomaly/diagnose`
- 输入字段建议：
  - `target_id`
  - `telemetry`
  - `network_metrics`
  - `context`
- 输出字段建议：
  - `root_cause`
  - `confidence`
  - `affected_scope`
  - `recommended_actions`

方案 C：时序评分接口

- `POST /api/anomaly/score`
- 输出字段建议：
  - `score`
  - `window_start`
  - `window_end`
  - `top_features`

前端只需要消费上述结果，不耦合具体算法框架。

## 6. 千问接入方案

### 6.1 接入原则

千问 API Key 不应硬编码在前端源码中，也不应直接写入可提交的版本文件。正确做法是通过本地环境变量让 Mock 服务或后端读取。

### 6.2 本项目采用的接入方式

本次实现将 Mock 服务的 LLM 调用改为支持 OpenAI 兼容模式，并优先读取以下环境变量：

- `SATOPS_COORDINATOR_API_KEY`
- `SATOPS_SPECIALIST_NETWORK_API_KEY`
- `SATOPS_LLM_API_KEY`
- `DASHSCOPE_API_KEY`

优先级从上到下回退。

### 6.3 默认模型与端点

默认配置为阿里云 DashScope 的 OpenAI 兼容模式：

- Base URL: `https://dashscope.aliyuncs.com/compatible-mode/v1`
- 调用路径：`/chat/completions`

模型建议：

- Coordinator: `qwen-plus`
- Network Specialist: `qwen-turbo`

后续可以在页面里改成你实际想用的模型。

### 6.4 本地配置方式

在项目根目录创建 `.env.mock`，写入：

```env
DASHSCOPE_API_KEY=你的千问Key
SATOPS_LLM_API_KEY=你的千问Key
```

如果希望为不同角色配置不同密钥，可以继续增加：

```env
SATOPS_COORDINATOR_API_KEY=你的协调器Key
SATOPS_SPECIALIST_NETWORK_API_KEY=你的网络专家Key
```

然后重启 `node mock-server.cjs`。

## 7. Mock 服务改造说明

为保证前端可先独立演示，本次实现会同步增强 `mock-server.cjs`：

- 增加 Qwen 兼容默认端点
- 支持 `DASHSCOPE_API_KEY` 作为回退读取
- 保留无密钥时的本地 mock 回答
- 保留 trace、blackboard、approval、A2A negotiation 的假数据闭环

这样可以实现：

- 有密钥时走真实大模型
- 无密钥时仍可演示前端功能

## 8. 前端实现细节

### 8.1 状态刷新

页面会定时轮询以下接口：

- Agent 状态
- LLM 状态
- 待审批项
- A2A 协商记录

轮询周期控制在 20 秒左右，兼顾展示实时性与接口压力。

### 8.2 消息流设计

聊天消息结构建议包含：

- `role`
- `content`
- `time`
- `agentType`
- `traceId`
- `suggestions`
- `actions`

这样既能展示自然语言结果，也能附带结构化建议。

### 8.3 Blackboard 展示

黑板区以键值表方式展示，重点字段包括：

- 当前网络状态
- 星座构型
- 日志指针
- 计划提示
- 已受理问题

### 8.4 审批展示

审批区突出显示：

- 动作类型
- 目标对象
- 风险级别
- 原因
- 创建时间

并支持快速通过/驳回。
